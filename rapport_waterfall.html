01 Introduction & Contexte du Projet

SafarAI est un concierge de voyage numérique dédié au Maroc. L'application est construite avec React/Vite pour le frontend et un orchestrateur IA agentique en backend qui utilise l'API Groq (modèle llama-3.3-70b-versatile) pour comprendre les requêtes des utilisateurs et y répondre de manière intelligente.

Le projet intègre plusieurs composants avancés :

  • Persona SafarAI : Une personnalité marocaine culturellement riche nommée « SafarAI » qui supporte le Darija, le Français et l'Anglais.
  • Moteur d'Intent : Une couche de raisonnement qui catégorise les messages en [EXPLORE, WEATHER, CULTURE, CHAT].
  • Tool Calling : L'API OpenTripMap est utilisée pour la recherche en temps réel de monuments et points d'intérêt.
  • Base de Connaissances : Un dossier safarai_docs/ contenant des guides touristiques officiels du Maroc.
📍 Contexte par défaut
Lors de cette session, le contexte par défaut est Ifrane, Maroc (Avril 2026), mais le système supporte désormais dynamiquement plus de 35 villes marocaines.

02 Problématique : Pourquoi Migrer ?

L'architecture précédente utilisait un appel LLM unique qui demandait au modèle de faire simultanément trois choses :

  1. Classifier l'intent de l'utilisateur (EXPLORE, CHAT, etc.)
  2. Décider et exécuter des appels d'outils (OpenTripMap)
  3. Formater la réponse en JSON structuré strict
⚠️ Problème Identifié
Faire les trois tâches en un seul appel causait des hallucinations fréquentes : le modèle inventait des arguments d'outils fictifs, cassait la structure JSON, ou mélangeait les formats de réponse. Ce comportement est connu sous le nom de « task interference » dans la littérature LLM.

La solution adoptée est la « 2-Step Waterfall Architecture » (en réalité 3 étapes), qui découple chaque responsabilité dans une étape séparée et déterministe.

03 Concepts Clés Expliqués

3.1 — Qu'est-ce qu'un LLM ? (Large Language Model)

🧠 Définition
Un LLM (Grand Modèle de Langage) est un réseau de neurones entraîné sur d'énormes quantités de texte. Il prédit le token suivant dans une séquence, ce qui lui permet de « comprendre » et « générer » du langage naturel. Dans notre cas, nous utilisons Llama 3.3 70B de Meta, hébergé via l'API Groq.

Le modèle reçoit un prompt (instructions + contexte) et génère une réponse texte. La qualité de la réponse dépend directement de la clarté et de la spécificité du prompt — c'est le principe du prompt engineering.

3.2 — Qu'est-ce que le Tool Calling ?

🔧 Définition
Le Tool Calling (appel d'outils) est une capacité des LLMs modernes qui leur permet de demander l'exécution de fonctions externes pendant la génération de leur réponse. Au lieu de deviner des données, le modèle peut appeler une API pour obtenir des informations réelles.

Exemple concret dans SafarAI : Quand un utilisateur demande « Que faire à Fès ? », le modèle reconnaît qu'il a besoin de données géographiques réelles. Il émet un appel search_opentripmap(lat=34.03, lon=-5.00, kinds="historic,cultural"). Le système exécute cet appel, récupère les résultats, et les injecte dans le contexte pour que le modèle formule une réponse enrichie avec des données factuelles.

3.3 — Qu'est-ce que l'Architecture Waterfall ?

🏗️ Définition
L'architecture Waterfall (cascade) est un modèle de conception où le traitement se fait en étapes séquentielles et indépendantes. Chaque étape reçoit l'output de la précédente et produit un résultat spécifique. Il n'y a pas de boucle récursive — les données « coulent » dans une seule direction, comme une cascade.

Pourquoi c'est supérieur pour les LLMs : Chaque étape donne au modèle une seule tâche claire à accomplir, ce qui réduit considérablement les hallucinations et augmente la fiabilité du système.

3.4 — Qu'est-ce que la Classification d'Intent ?

🎯 Définition
La classification d'intent (intention) consiste à déterminer automatiquement ce que l'utilisateur veut faire à partir de son message en langage naturel. C'est une tâche de NLU (Natural Language Understanding).

Dans SafarAI, nous utilisons 4 catégories d'intent :

Intent Description Exemple Action
EXPLORE L'utilisateur veut découvrir des lieux « Que faire à Ifrane ? » → Appel OpenTripMap
WEATHER Question sur la météo « Quel temps fait-il à Casa ? » → Appel OpenTripMap + météo
CULTURE Question culturelle sur le Maroc « C'est quoi le zellige ? » → Réponse directe du LLM
CHAT Conversation générale « Salam ! Kidayr ? » → Réponse directe du LLM

3.5 — Qu'est-ce que l'API OpenTripMap ?

🗺️ Définition
OpenTripMap est une API REST gratuite qui fournit des données géographiques sur les points d'intérêt touristiques. En lui envoyant des coordonnées GPS (latitude/longitude) et des catégories (historique, culturel, naturel...), elle retourne une liste de lieux réels avec leurs noms et descriptions.

URL de l'appel :
https://api.opentripmap.com/0.1/en/places/radius?radius=10000&lon={lon}&lat={lat}&kinds={kinds}

JSON → UI Cartes, météo, itinéraire
Étape Rôle LLM ? Outils API ? Température
Stage 1 Classifier l'intent + extraire les entités (ville, catégories) ✅ Oui ❌ Non 0.1 (déterministe)
Stage 2 Résoudre les coordonnées + appeler OpenTripMap ❌ Non ✅ Oui —

💡 Avantage clé
Chaque étape a une seule responsabilité. Le LLM de l'Étape 1 ne fait que classifier — il n'a pas à se soucier de la mise en forme. Le LLM de l'Étape 3 ne fait que rédiger — il n'a pas à décider quels outils appeler. Cette séparation élimine l'interférence entre tâches.

05 Étape 1 : Classification d'Intent

Cette étape envoie un appel LLM léger et ciblé. Les paramètres clés :

  • Température : 0.1 — Très basse pour garantir des résultats déterministes
  • max_tokens : 200 — La réponse attendue est un petit objet JSON
  • Aucun outil — Le LLM se concentre uniquement sur la classification

Prompt système (simplifié) :

// Le LLM reçoit ce prompt système :
"Vous êtes un classificateur d'intent pour SafarAI.
Analysez le message et retournez un JSON :
{
  intent: EXPLORE | WEATHER | CULTURE | CHAT,
  city: 'nom de la ville ou null',
  lat: number ou null,
  lon: number ou null,
  kinds: 'catégories OpenTripMap ou null'
}"

Code source :

const analyzeIntent = async (userInput) => {
  console.log('[Waterfall] Stage 1: Analyzing intent...');

  const payload = {
    model: MODEL_NAME,
    messages: [
      { role: 'system', content: INTENT_PROMPT },
      { role: 'user', content: userInput }
    ],
    temperature: 0.1,   // Faible → déterministe
    max_tokens: 200      // Réponse courte attendue
  };

  const message = await callGroq(payload);
  const result = JSON.parse(message.content);

  return {
    intent: result.intent || 'CHAT',
    city: result.city || null,
    kinds: result.kinds || 'interesting_places'
  };
};
🔑 Concept : Température du LLM
La température contrôle le caractère aléatoire de la génération. 0.0 = toujours la même réponse (déterministe). 1.0 = réponses variées et créatives. Pour la classification d'intent, une température de 0.1 est idéale car on veut une catégorisation fiable et reproductible, pas de la créativité.

06 Étape 2 : Exécution des Outils

C'est l'étape purement déterministe — aucun LLM n'est impliqué. Le code JavaScript natif fait deux choses :

  1. Résoudre les coordonnées : Convertir le nom de la ville en latitude/longitude via une table de correspondance
  2. Appeler l'API : Envoyer une requête à OpenTripMap avec les coordonnées résolues

Logique de décision :

const executeTools = async (intentResult) => {
  // Seuls EXPLORE et WEATHER déclenchent un appel d'outil
  if (intentResult.intent !== 'EXPLORE' &&
      intentResult.intent !== 'WEATHER') {
    return null;  // CHAT et CULTURE → pas d'outil
  }

  // Résolution : ville → coordonnées → API
  let coords = resolveCityCoords(intentResult.city);
  if (!coords) coords = DEFAULT_COORDS; // Fallback: Ifrane

  return await fetchOpenTripMapData(
    coords.lat, coords.lon, intentResult.kinds
  );
};
💡 Concept : Exécution Déterministe vs. Probabiliste
Quand on utilise un LLM pour décider quels outils appeler, le résultat est probabiliste — il peut varier d'un appel à l'autre. En déplaçant cette logique dans du code JavaScript natif (conditions if/else), le comportement devient déterministe : le même input produira toujours le même output. C'est la raison principale de la migration.

07 Étape 3 : Génération de la Réponse Finale

C'est le deuxième (et dernier) appel LLM du pipeline. Il reçoit :

  • Le message original de l'utilisateur
  • L'intent classifié (de l'Étape 1)
  • Les données d'outils (de l'Étape 2), pré-injectées dans le prompt système

Critiquement, cet appel n'a aucun outil attaché. Le modèle se concentre à 100% sur la rédaction de la réponse dans le format JSON attendu, avec la personnalité marocaine authentique de SafarAI.

Structure JSON attendue en sortie :

{
  "intent": "recommendation",
  "text": "**Ifrane, la Suisse du Maroc** 🏔️ ...",
  "destination": "Ifrane",
  "weather": { "condition": "Ensoleillé", "temperature": "18°C" },
  "monuments": [
    { "name": "Al Akhawayn", "description": "..." }
  ],
  "itinerary": [],
  "suggestions": ["Visiter le Lion de Pierre"]
}
📐 Concept : Injection de Contexte
Au lieu de demander au LLM de chercher lui-même les données (ce qui mène à des hallucinations), on lui fournit les données réelles dans le prompt. Le LLM n'a plus qu'à les reformuler de manière conversationnelle. C'est le pattern de Retrieval-Augmented Generation (RAG).

09 Parser JSON Robuste

Un des défis techniques majeurs était le parsing de la réponse JSON du LLM. Les LLMs ont tendance à envelopper leur réponse de différentes manières :

Problème Exemple Solution
Code fences Markdown ```json { ... } ``` Regex pour extraire le contenu entre les fences
Enveloppe de schéma { type: "JSON", schema: { ... } } Détection et « unwrapping » automatique
Texte autour du JSON Voici la réponse : { ... } Regex /\{[\s\S]*\}/ pour extraire

Code du parser :

export const parseResponse = (rawText) => {
  // 1. Supprimer les code fences Markdown
  let cleaned = rawText;
  const fenceMatch = cleaned.match(/```(?:json)?\s*([\s\S]*?)```/);
  if (fenceMatch) cleaned = fenceMatch[1].trim();

  // 2. Extraire le JSON
  const jsonMatch = cleaned.match(/\{[\s\S]*\}/);
  let parsed = JSON.parse(jsonMatch[0]);

  // 3. Déballer l'enveloppe de schéma si présente
  if (parsed.schema && !parsed.text) {
    parsed = parsed.schema;
  }

  return parsed;
};

12 Conclusion

La migration vers l'architecture Waterfall a résolu le problème principal de SafarAI : les hallucinations et les erreurs de format JSON causées par la surcharge cognitive du LLM.
